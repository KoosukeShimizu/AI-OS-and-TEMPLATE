# 0. このOSの存在目的（Why）

このOSは以下のために存在する：

- 構造の安定を守るため
- 人間とAIの思考を同期させるため
- 局所最適ではなく全体最適を実現するため
- 一時的な正解ではなく「拡張可能な正解」を作るため

---

# 1. 最上位判断原則（迷ったらここを見る）

迷った場合、以下を自問する：

- これは構造を壊していないか？
- 将来拡張できる設計か？
- 単発解決になっていないか？
- 再利用可能な形になっているか？
- ユーザーは自己解決できるか？

反する場合、原則として修正する。

# 2. 設計思想レイヤー（How）

## 業務ドリブン優先

- UIは装飾ではなく業務達成のために存在する
- 目的達成に直接関係しない装飾は削る

## 自己解決可能UX

- マニュアル依存を減らす
- 状態を明確にする（loading / empty / error）
- ユーザーが迷わない情報構造を作る

## 情報は整理されていること

- 情報過多は設計の失敗
- 優先順位を必ず明示する
- L1 / L2 / L3構造を意識する

## ごちゃごちゃは古い

- 情報の密度は制御する
- 余白は意味を持つ
- ノイズを削る

## 空気を設計する（デザイン方針）

- レイアウトとタイポで空気感を作る
- 余白で緊張と緩和を作る
- 分割と非対称でリズムを作る
- ベースは静かに、1点だけ強く

## トレンドは吸収するが依存しない

- レイアウトトレンドは積極的に取り入れる
- カラーはニュートラルベースを軸にする（くすみ・無彩色寄り）
- UIエフェクト（ブラー/ガラス等）は必要な時だけ使う
- 流行は“構造に乗せる”のであって、構造を壊さない

## 静かに尖る（表現方針）

- 全体は静かに整える
- しかし必ず一点に強度を持たせる
- 派手さではなく、構造と余白で差を出す
- 過剰に主張しない
- 気づいた人に刺さる設計を目指す

---

# 3. 優先順位（トレードオフ時）

1. 再利用性
2. 拡張性
3. 可読性
4. 実装効率
5. 表現の派手さ

※ 例外がある場合は ai_context.md に明記する

---

# 4. 技術より構造

- フレームワークは道具である
- 技術選定は思想に従う
- 「書ける」より「維持できる」を優先する

---

# 5. 局所解の禁止

以下は禁止に近い扱いとする：

- 単発UIの乱立
- 状態管理を考えない実装
- 後から説明できない命名
- その場しのぎのif分岐

---

# 6. 自由にしてよい範囲（他人に渡す時のため）

- 「静かに尖る」の“一点”は案件ごとに変えてよい
  - タイポ強度 / 余白 / 分割 / 色アクセントのどれか1つ
- ただし、尖りを増やすために構造を壊してはいけない
- “派手さの追加”は最後の手段であり、最初の解ではない
