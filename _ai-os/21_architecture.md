# 21_architecture.md

## 参照タイミング
以下に該当する場合は必ず参照する（構造設計の判断が発生するタイミング）：

- ページ/画面のセクション分解（L1/L2/L3）を作る・変更する
- コンポーネント抽出（再利用化）を行う
- include分割粒度・責務の切り分けを決める
- 既存構造の改変を伴う可能性がある（過剰ネストの解消等）

## このファイルについて
画面・ページ・コンポーネント設計の骨格を定義する。
AIは構造判断を行う際、このルールを最優先で参照すること。

# 1. 基本設計思想

- コンポーネント単位で分解する
- 横断利用前提で設計する
- 単発UIを作らない
- L1 / L2 / L3 レイヤー構造を意識する
- ページから構造が把握できる粒度を保つ（過剰ネスト禁止）
- Modeは判断軸であり、構造そのものは変更しない（24に従う）

---

# 2. L1 / L2 / L3 構造

## L1：状況把握レイヤー
- 通知
- ステータス
- 現在位置
- パンくず
- ページタイトル

## L2：業務・主機能レイヤー
- フォーム
- テーブル
- コンテンツ本文
- 入力UI
- 主要アクション

## L3：補助レイヤー
- ヘルプ
- 注意文
- 補足リンク
- フッター情報

構造設計時はこの3層を必ず意識する。

---

# 3. Pug構造原則

## 3-1. extendsは禁止（常に）

- layout.pugは使用しない
- 継承構造は作らない
- includeで明示的に読み込む

理由：
構造の可視性を保つため。

---

## 3-2. include粒度（中規模以上）

includeの置き場は [ai/23_environment.md](ai/23_environment.md)、許可粒度は [ai/40_markup_pug.md](ai/40_markup_pug.md) を参照する。

---

## 3-3. control block（中規模以上の全ページ必須）

中規模以上の全ページに以下を持つ（LPは 40_markup_pug.md の LP-MIN 定義に従う）：

必須：
- key（ナビcurrent制御用）
- slug
- parentSlug（空でも保持）
- version（キャッシュクリア用）

任意：
- pattern（A/B等）
- state（フェーズ差分）
- cpEnd（キャンペーン終了制御）

---

## 3-4. key と slug の役割分離

- key：ナビcurrent制御用（form/confirm/thanks を同一制御にできる）
- slug：パンくず用
- parentSlug：パンくず階層制御用

ナビとパンくずはロジックを分離する。

---

# 4. コンポーネント分類（設計時の共通言語）

## Foundation
- reset / base / token 等
- 影響範囲が広いので慎重に変更

## Layout
- ヘッダー / フッター / ラッパー / ページ枠

## Component
- 再利用前提のUI部品

## Project
- ページ専用構造
- Component化を検討した上で最後に置く

## Utility
- 1責任1プロパティ
- 乱立禁止

---

# 5. 単発構造禁止

- そのページでしか使わない構造は
  一度コンポーネント化可能性を検討する
- Speed-firstでも「最小限の再利用可能形」は守る

---

# 6. 状態設計

該当する場合は必ず設計する：

- loading
- empty
- error

Data-denseモードでは必須。

---

# 7. 差分改修原則（Legacy-safe）

- 既存構造を壊さない
- 追加で解決できないか検討する
- 全面置換は禁止
- 命名統一は新規部分から段階的に行う

---

# 8. 構造チェック項目

- L1/L2/L3が整理されているか
- control blockが存在するか
- include粒度が過剰でないか
- ページから構造が読めるか
- 単発UIが増殖していないか
- 既存改修で破壊的変更になっていないか
