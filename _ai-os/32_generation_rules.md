# 32_generation_rules.md

## 参照タイミング
常時参照（実装出力の安全装置。出力前に必ず適用する）。

## このファイルについて
**このファイルは AI-OS における「実装時の唯一の強制ルール」である。**  
AIが実装作業を行う際、すべての出力はここに定義したルールに従わなければならない。  
40〜42（実装ガイド）や 50〜53（レビュー/監査）と役割が異なり、  
それらよりも常に優先される。  

構造破壊・推測削除・全面置換を防ぐための安全装置である。

Updated: 2026-04-16
---

## Enforcement

本ファイルのルールは「絶対ルール」である。

以下の場合、必ず本ファイルを優先すること：

- Figmaデザインと矛盾する場合
- テンプレ構造と矛盾する場合
- 一般的な実装慣習と異なる場合
- 40〜42（実装ガイド）の記述と解釈が衝突する場合
- 50〜53（レビュー/監査）の指摘と実装判断が衝突する場合

本ファイルに反する実装はすべて禁止とする。  
違反の可能性がある場合は、実装を止めて確認を行うこと。

---

## Violation Handling

本ルールに違反する可能性がある場合：

- 実装を一時停止する
- ai_context.md に未確定事項として記録する
- 必要に応じて ai_decision_log.md に記録する
- ユーザーに確認する

推測でルールを破ってはいけない。

---

## Allowed Flexibility

以下はルール違反とみなさない：

- 軽微なマークアップ調整（wrapper追加など）
- CSSで吸収可能な構造差
- 既存構造を壊さない範囲での最適化
- 可読性向上のためのリファクタ

ただし「構造変更」と「設計変更」は禁止対象とする。

---

## 実行環境前提

本OSは「MCP対応AIエージェント」を前提とする。

想定環境：
- Claude Code
- VSCode + MCP
- 同等のプロジェクト横断編集が可能なAI

AIは以下を前提とする：

- プロジェクト全体を参照可能
- 複数ファイル横断編集が可能
- Figma MCP等の外部参照が可能

ただし、特定ベンダー依存の設計は行わない。

# 1. 実装前の確認（必須）

実装前に必ず確認する：

- ai_context.md を参照したか
- Modeを理解しているか（31_design_modes.md）
- ai_outline.md が確定しているか
- 影響範囲（どのファイル/どのコンポ/どのページ）が明確か
- Todoが必要か判定したか（10_workflow.md参照）

未確定事項がある場合は、実装を止めて質問する。

---

# 2. 出力は「差分・追加単位」

- 既存ファイルを全面置換しない
- 変更する場合は「変更前 / 変更後」を明示する
- 追加する場合は「挿入位置（どこに入れるか）」を明示する
- 削除が発生する場合は理由と依存確認を明示する（推測削除は禁止）

---

# 3. 実装の順序（固定）

1. Pug（構造/骨格）
2. SCSS（最低限→調整）
3. JS（初期化→詳細）

順序を飛ばさない。

---

# 4. Todo連動ルール（必須）

以下のいずれかに該当する場合、Todoリストを作成する（10_workflow.md準拠）：

- 2ファイル以上に変更が跨る
- 2コンポ以上に影響する
- 状態が3種以上ある（loading / empty / error 等）
- 外部ライブラリ導入・設定が発生する
- 既存改修で影響範囲が不明瞭

Todoは最大7項目まで。  
Yes / No で完了判定できる粒度で作成する。

---

# 5. 推測禁止

- 既存仕様を推測しない
- 未確認の削除を行わない
- デザイン意図を勝手に補完しない
- 仕様が曖昧な場合は質問する

---

# 6. ライブラリ導入時（必須）

導入する場合は必ず明記する：

- なぜ必要か
- 影響範囲（ページ/コンポ）
- CSS依存の有無
- 初期化方法
- data-*トリガーで起動できるか（できない場合は理由）

---

# 7. 破壊的変更の禁止

- layout.pugの導入（extends禁止）
- ディレクトリ再構築
- 命名体系の全面変更
- __dist の直接編集
- 一括置換の提案（差分で提示する）

必要な場合は decision_log対象とする。

---

## 7-1. 編集境界ルール（最重要・常時適用）

AIが直接編集してよい対象は、原則として以下に限定する：

- pug/
- scss/
- script/（JSモジュール）
- ai-runtime/（生成ファイルのみ）

以下は原則編集禁止：

- __dist/
- build成果物
- node_modules/
- package.json（明示的指示がない限り）
- .htaccess（明示的指示がない限り）
- .htpasswd（明示的指示がない限り）
- prepros.config（明示的指示がない限り）
- .env（明示的指示がない限り）
- robots.txt（明示的指示がない限り）
- 設定ファイル（vite.config / webpack / tsconfig 等）
- ディレクトリ構造そのものの再編成

### ディレクトリ構造の唯一の正

SSoT（唯一の正）の定義は 01_os_lock.md を参照する。

### 禁止事項（明示）

- 新規トップレベルフォルダの作成
- 既存フォルダ名の変更
- include構造の再設計
- ビルド出力物の直接編集
- “とりあえず動かすための仮置きファイル”の生成

必要な場合は必ず decision_log 対象とする。

# 8. 追加の安全チェック（実務ルール）

## 8-1. control block確認（Pug修正時）

Pug修正時は必ず確認：

- key は意図どおりか（current制御の単位として妥当か）
- slug は適切か
- parentSlug は親がない場合でも空で保持されているか
- version は存在するか（キャッシュクリア用）

---

## 8-2. JS出力ルール（実務）

- data-*トリガーを使用する（可能な限り100%）
- DOMContentLoadedを重複させない
- module分割を守る（script/_module）
- main.jsの初期化順を壊さない
- 見た目はCSS、JSは挙動（class付与/状態制御まで）

---

## 8-3. SCSS出力ルール（実務）

- _foundation を安易に書き換えない（影響が大きい）
- _component にページ専用スタイルを書かない（_projectへ）
- _utility を乱立させない
- import順を崩さない（style.scssの順序を尊重）

---

## 8-4. 改修案件ルール（重要）

- 既存命名を壊さない
- 既存classを推測で削除しない
- 差分範囲を明示する
- 既存ファイル全置換は禁止

---

## 8-5. Figma参照時ルール（重要）

ai_context.md 内に「Figma参照あり」が明記されている場合に適用する。

- Figmaは「レイアウト参照ソース」として扱う（完全一致を前提にしない）
- レイヤー名 / フレーム名をそのままHTMLクラス名に採用しない
- HTML構造・クラス命名はプロジェクト定義（命名規約/既存コンポ）を最優先とする
- 変換は「ページ全体」ではなく「セクション単位」で進める（差分運用）
- まず骨格（セクション/主要ブロック/見出し階層）を揃え、その後にパーツ単位で精度を上げる
- Figmaに存在しない要素（エラーパターン/空状態/hover等）を推測で追加しない
- 画像・テキストが不足している場合はダミーで埋め、未確定事項に明記する
- 余白・サイズ・行間はCSSで吸収する（HTMLを過度にネストして合わせない）
- 位置合わせ目的の空div・br乱用は禁止（必要な場合は理由を明記）
- フォント/色/影などのトークン化は「今回のスコープ」に含まれている時だけ行う（推測で設計しない）

---

## 8-6. デザインシステム参照時ルール（DS駆動モード、重要）

プロジェクトルートに `design-system/` ディレクトリが存在する、または ai_context.md に「Figma DS 駆動」が明記されている場合に適用する。

### 強制事項

- 色・spacing・radius・shadow・typography は **Theme トークン経由でのみ出力**（生hex / 生px / $c-* 直参照は禁止）
- トークン名の変換は `63_design_tokens_output.md` の leaf ルールを唯一の正とする（Figma階層 `Theme/NN_/` を出力に含めない）
- `_Reserved/` 配下 / `⚠_` prefix の Variable は出力対象外（`65_figma_constraints.md` 参照）
- `design-system/decisions.md` に記録された DS 固有判断は、40〜42（実装ガイド）の一般規約より優先する

### 既存 SCSS トークンとの衝突時

- 新規案件（design-system/ を初期状態で同梱）：Figma Theme を正、既存 `_config.scss` / `_tokens.scss` は Figma Theme で置換可
- 既存改修案件（旧 `_tokens.scss` が運用中で後から design-system/ を導入）：**推測で rename しない**。`design-system/decisions.md` に移行方針を記録してから着手
- 命名が衝突する場合（例：旧 `--bg-base` vs 新 `--bg-default`）は、必ず `decisions.md` に対応表を残す

### 禁止事項

- Figma DS にある値を SCSS 側で独自に定義し直す（値の二重管理を作る）
- `decisions.md` に記録なく旧トークンを削除する
- AI の判断で新旧どちらの命名が正かを推測決定する

### 未確定時

DS 駆動プロジェクトで命名対応が不明な場合、実装を止めて `ai_context.md` に未確定事項として記録し、ユーザーに確認する。

# 9. AIの最終自己確認

出力前に自問する：

- これは差分になっているか？
- 既存を壊していないか？
- Modeに沿っているか？
- 将来拡張できる構造か？
- 32_generation_rules.md に違反していないか？
- DS 駆動プロジェクトの場合、生値でなく Theme トークン経由で出力しているか？（8-6）
